## Overview

Address critical safety issues, reduce ~2K lines of duplicated CRUD boilerplate, add request body validation, and clean up miscellaneous issues discovered during the architectural review of the dual-backend EMR implementation.

## Current State Analysis

The `tabs-settings-clean-up` branch has a working dual-backend system (local PostgreSQL vs Medplum FHIR) with 19 clinical API routes, 18 FHIR service files, and a runtime toggle mechanism. The architectural review identified 13 issues across 3 severity levels (3 critical, 6 important, 6 suggestion).

### Key Discoveries:
- 12 of 19 clinical routes are structurally identical ŌĆö only model name and field tuple differ ([allergy/route.ts](app/api/clinical/allergy/route.ts), [condition/route.ts](app/api/clinical/condition/route.ts), etc.)
- Clinical routes have no `isLocalBackend()` guard ŌĆö they will crash if accessed in Medplum mode ([allergy/route.ts:28](app/api/clinical/allergy/route.ts#L28) calls `prismaClinical` unconditionally)
- `instrumentation.ts` has no try/catch around `loadEmrBackendFromDb()` ŌĆö DB failure crashes startup ([instrumentation.ts:6](instrumentation.ts#L6))
- `patient-tags/route.ts` writes `patientFhirId: ""` in local mode, violating `@@unique([patientFhirId, category, value])` ([patient-tags/route.ts:92](app/api/patient-tags/route.ts#L92))
- `admin/settings/route.ts` mutates `process.env` at runtime, which is broken in serverless and has no effect on `NEXT_PUBLIC_*` vars ([admin/settings/route.ts:51](app/api/admin/settings/route.ts#L51))
- Generated Prisma client (~40K lines) is in `.gitignore` but still tracked by git
- `case-library/route.ts` duplicates `patient-id.ts` logic with its own `getClinicalPatientId` function ([case-library/route.ts:15](app/api/case-library/route.ts#L15))

## Desired End State

After this plan:
1. Clinical routes return 501 when accessed in Medplum mode (preventing crashes)
2. `instrumentation.ts` handles DB failures gracefully (falls back to env var)
3. `patientFhirId` is `null` (not `""`) in local mode for patient tags
4. 12 standard clinical routes are generated by a factory function (~120 lines replacing ~2K)
5. All clinical route inputs are validated with Zod schemas
6. Generated Prisma client is untracked from git
7. Runtime `process.env` mutation has a clear comment about its limitations
8. `case-library/route.ts` uses shared `patient-id.ts` utilities

### Verification:
- `npx tsc --noEmit` passes
- `npm run lint` passes
- `npm run build` succeeds
- Manual test: clinical routes return 501 JSON error in Medplum mode
- Manual test: clinical CRUD works in local mode
- Manual test: patient tags work without unique constraint violations

## What We're NOT Doing

- **Strategy/adapter pattern** for backend abstraction ŌĆö the `isLocalBackendClient()` branching in services is working and changing the pattern is a larger refactor
- **DateTime migration** ŌĆö storing dates as `String` works; migrating requires data migration and is a separate effort
- **Pagination** on list endpoints ŌĆö not needed at current data volumes
- **`pick()` return type improvement** ŌĆö would require generic overloads and doesn't affect runtime behavior
- **Missing DELETE handlers** (encounter, vital, task, medication) ŌĆö can be added later as needed
- **`any` types in FHIR service local branches** ŌĆö typing improvement, not a safety issue
- **`communication` and `goal` route standardization** ŌĆö their non-standard patterns are intentional (dual-entity management, coding object reshaping)

## Implementation Approach

Four phases, ordered by impact and dependency:
1. **Critical safety fixes** ŌĆö prevent crashes and data corruption (standalone, no dependencies)
2. **CRUD route factory** ŌĆö extract shared pattern, migrate 12 routes (standalone, guard handled by middleware)
3. **Zod validation** ŌĆö add input validation to factory and non-standard routes (depends on Phase 2 factory)
4. **Cleanup** ŌĆö git hygiene, consolidate duplicated code (standalone)

## Phase 1: Critical Safety Fixes

### Overview
Fix the three critical issues that can cause crashes or data corruption.

### Changes Required:

#### 1. Add backend guard via Next.js Middleware

**File**: [middleware.ts](middleware.ts)

Add a clinical route guard in the existing middleware. This is fail-safe ŌĆö any new `/api/clinical/*` route is automatically protected without developers needing to remember a per-route guard.

Add the following block **after** the public routes check and **before** the auth check:

```typescript
// Clinical API routes are local-backend-only.
// Block them in Medplum mode to prevent prismaClinical crashes.
if (pathname.startsWith("/api/clinical/")) {
  const backend = process.env.EMR_BACKEND?.toLowerCase()
  if (backend !== "local") {
    return NextResponse.json(
      { error: "Clinical API is not available in Medplum mode" },
      { status: 501 }
    )
  }
}
```

**Note**: We read `process.env.EMR_BACKEND` directly instead of importing `isLocalBackend()` to avoid pulling in module dependencies that may not be Edge-compatible. The env var is set at startup by `instrumentation.ts`.

No changes needed to individual route files for the guard ŌĆö the middleware handles it globally.

#### 2. Wrap instrumentation.ts in try/catch
**File**: [instrumentation.ts](instrumentation.ts)

```typescript
export async function register() {
  if (process.env.NEXT_RUNTIME === "nodejs") {
    try {
      const { loadEmrBackendFromDb } = await import("@/lib/app-config")
      await loadEmrBackendFromDb()
    } catch (error) {
```

**Note**: Uses `console.warn` instead of `logger` because `logger.ts` may have its own module-level dependencies that could fail during early startup.

## Phase 2: CRUD Route Factory

### Overview
Extract the shared CRUD pattern from 12 identical clinical routes into a factory function, reducing ~2K lines to ~120.

### Changes Required:

#### 1. Create the factory function
**File**: `lib/clinical-crud-factory.ts` (new)

```typescript
import { NextRequest, NextResponse } from "next/server"
import { requireAuth, isSession } from "@/lib/auth-helpers"
import { prismaClinical } from "@/lib/prisma-clinical"
import { requirePatientAccess } from "@/lib/clinical-auth"
import { pick } from "@/lib/pick"
import { logger } from "@/lib/logger"
import type { Prisma } from "@/app/generated/prisma-clinical"

/** Lowercase model names available on prismaClinical */
type ClinicalModelName = Uncapitalize<Prisma.ModelName>

interface CrudConfig {
  /** Prisma model name (type-checked against generated client) */
  model: ClinicalModelName
  /** Allowlisted fields for create/update */
  fields: readonly string[]
  /** Human-readable resource name for error messages */
  label: string
}

/**
 * Creates standard GET/POST/PUT/DELETE route handlers for a clinical resource.
 *
 * Backend guard is handled by middleware.ts (blocks /api/clinical/* in Medplum mode),
 * so individual handlers don't need to check the backend.
 */
export function createClinicalCrudHandlers(config: CrudConfig) {
  const { model, fields, label } = config
  // eslint-disable-next-line @typescript-eslint/no-explicit-any
  const delegate = (prismaClinical as any)[model]

  async function GET(request: NextRequest) {
    const authResult = await requireAuth()
    if (!isSession(authResult)) return authResult

    try {
      const patientId = request.nextUrl.searchParams.get("patient")
      if (!patientId) {
        return NextResponse.json({ error: "patient parameter is required" }, { status: 400 })
      }

      const forbidden = await requirePatientAccess(authResult, patientId)
      if (forbidden) return forbidden

      const items = await delegate.findMany({
        where: { patientId },
        orderBy: { createdAt: "desc" },
      })

      return NextResponse.json({ items, total: items.length })
    } catch (error) {
      logger.error(`Clinical ${label} search error`, error)
      return NextResponse.json({ error: `Failed to fetch ${label}` }, { status: 500 })
    }
  }

  async function POST(request: NextRequest) {
    const authResult = await requireAuth()
    if (!isSession(authResult)) return authResult

    try {
      const body = await request.json()
      if (!body.patientId) {
        return NextResponse.json({ error: "patientId is required" }, { status: 400 })
      }

      const forbidden = await requirePatientAccess(authResult, body.patientId)
      if (forbidden) return forbidden

      const data = pick(body, fields)
      const item = await delegate.create({ data })
      return NextResponse.json({ id: item.id })
    } catch (error) {
      logger.error(`Clinical ${label} create error`, error)
      return NextResponse.json({ error: `Failed to create ${label}` }, { status: 500 })
    }
  }

  async function PUT(request: NextRequest) {
    const authResult = await requireAuth()
    if (!isSession(authResult)) return authResult

    try {
      const body = await request.json()
      const { id } = body

      if (!id) {
        return NextResponse.json({ error: `${label} ID is required` }, { status: 400 })
      }

      const existing = await delegate.findUnique({ where: { id }, select: { patientId: true } })
      if (!existing) {
        return NextResponse.json({ error: "Not found" }, { status: 404 })
      }

      const forbidden = await requirePatientAccess(authResult, existing.patientId)
      if (forbidden) return forbidden

      const data = pick(body, fields.filter((f: string) => f !== "patientId"))
      const item = await delegate.update({ where: { id }, data })
      return NextResponse.json(item)
    } catch (error) {
      logger.error(`Clinical ${label} update error`, error)
      return NextResponse.json({ error: `Failed to update ${label}` }, { status: 500 })
    }
  }

  async function DELETE(request: NextRequest) {
    const authResult = await requireAuth()
    if (!isSession(authResult)) return authResult

    try {
      const id = request.nextUrl.searchParams.get("id")
      if (!id) {
        return NextResponse.json({ error: "id parameter is required" }, { status: 400 })
      }

      const existing = await delegate.findUnique({ where: { id }, select: { patientId: true } })
      if (!existing) {
        return NextResponse.json({ error: "Not found" }, { status: 404 })
      }

      const forbidden = await requirePatientAccess(authResult, existing.patientId)
      if (forbidden) return forbidden

      await delegate.delete({ where: { id } })
      return NextResponse.json({ success: true })
    } catch (error) {
      logger.error(`Clinical ${label} delete error`, error)
      return NextResponse.json({ error: `Failed to delete ${label}` }, { status: 500 })
    }
  }

  return { GET, POST, PUT, DELETE }
}
```

**Key differences from original per-route approach:**

- **No `requireLocalBackend()` calls** ŌĆö middleware handles this globally for all `/api/clinical/*` routes
- **`model` is type-checked** via `ClinicalModelName` (derived from `Prisma.ModelName`) ŌĆö typos are caught at compile time

## Phase 3: Request Body Validation (Zod)

### Overview
Add Zod validation schemas for POST/PUT request bodies in both the factory and non-standard routes.

### Changes Required:

#### 1. Add Zod schemas to the factory
**File**: `lib/clinical-crud-factory.ts` (modify)

Add a `schema` field to `CrudConfig`:

```typescript
import { z } from "zod"

interface CrudConfig {
  model: string
  fields: readonly string[]
  label: string
  /** Zod schema for POST body validation. PUT schema is derived (all fields optional + id required). */
  schema: z.ZodObject<any>
}
```

Update POST handler to validate:
```typescript
async function POST(request: NextRequest) {
  // ... auth checks ...
  const body = await request.json()
  const parsed = config.schema.safeParse(body)
  if (!parsed.success) {
    return NextResponse.json({ error: "Validation failed", details: parsed.error.flatten().fieldErrors }, { status: 400 })
  }
  // ... use parsed.data instead of pick(body, fields) ...
}
```

Update PUT handler similarly, using a partial version of the schema with `id` required:
```typescript
async function PUT(request: NextRequest) {
  // ... auth checks ...
  const body = await request.json()
  const updateSchema = config.schema.partial().extend({ id: z.string().min(1) })
  const parsed = updateSchema.safeParse(body)
  if (!parsed.success) {
    return NextResponse.json({ error: "Validation failed", details: parsed.error.flatten().fieldErrors }, { status: 400 })
  }
  // ... use parsed.data (minus patientId) ...
}
```

With Zod validation, `pick()` is no longer needed in the factory ŌĆö `parsed.data` already contains only declared fields.

```typescript
/**
 * Get the clinical patient ID from a UserPatient record, with fallback.
 * Uses the backend-appropriate field, falling back to whichever ID is available
 * (handles records created under one backend but read under another).
 */
export function getClinicalPatientId(
  assignment: { patientFhirId: string; patientLocalId: string | null }
): string {
  const field = getPatientIdField()
  const value = assignment[field]
  if (value) return value
  return assignment.patientLocalId || assignment.patientFhirId
```
